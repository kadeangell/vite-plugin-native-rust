# TypeScript

Out of the box, `tsc` rejects `import { fn } from "./native/src/lib.rs"` ā€” it has
no idea what a `.rs` module exports. This plugin makes the import fully typed,
using real napi-generated types (not `any`), and keeps CI type-checking working
without a Rust toolchain installed.

## `allowArbitraryExtensions`

TypeScript 5.0+ can resolve types for an import with a non-standard extension via
a sibling declaration file. For `./lib.rs`, it looks for `./lib.d.rs.ts`. That
lookup is gated behind one compiler option:

```jsonc
// tsconfig.json
{
  "compilerOptions": {
    "allowArbitraryExtensions": true
  }
}
```

The plugin adds this for you automatically **if** your root `tsconfig.json` is
plain JSON it can safely parse. If your tsconfig uses JSONC (comments,
trailing commas) the plugin leaves it untouched to avoid clobbering it ā€” add the
option by hand.

## The generated `.d.rs.ts`

`@napi-rs/cli` emits an `index.d.ts` describing your crate's exports during every
build. After each successful compile, the plugin copies that file to
`<anchor>.d.rs.ts` right next to your imported `.rs` ā€” so `./lib.rs` gets a
`./lib.d.rs.ts` sibling. These are **real** types derived from your Rust
signatures:

```ts
// native/src/lib.d.rs.ts  (generated)
/* auto-generated by NAPI-RS */
/* eslint-disable */
export declare function add(a: number, b: number): number
export declare function hashChain(iterations: number): Promise<string>
```

Note the faithful mapping: Rust `i32` ā†’ `number`, `u32` ā†’ `number`, an
`async fn -> String` ā†’ `Promise<string>`, and `snake_case` ā†’ `camelCase`
(`hash_chain` becomes `hashChain`). Rust doc-comments carry through to the
`.d.ts`, so they show up on hover in your editor.

Disable the mirroring with `emitTypes: false` if you'd rather manage the
declaration yourself.

## Commit the `.d.rs.ts` (for CI)

**Commit the generated `.d.rs.ts` to source control.** Your CI type-check job
almost certainly runs `tsc` without installing a Rust toolchain ā€” and without the
committed declaration, `tsc` would have nothing to resolve the `.rs` import
against. The committed file is what makes `tsc --noEmit` pass in CI. It is
regenerated on every local compile, so it stays in sync as your Rust signatures
change; just commit the update alongside the Rust change.

The example app ships exactly this: `examples/react-router/native/src/lib.d.rs.ts`
is committed, and the app's `typecheck` script runs against it toolchain-free.

## The types are real ā€” a wrong-arity proof

Because the declaration is generated from the actual napi signature, TypeScript
catches misuse. Given the `add(a: number, b: number): number` above:

```ts
import { add } from "./native/src/lib.rs";

add(2, 3);        // ok
add(2);           // TS error: Expected 2 arguments, but got 1.
add("2", 3);      // TS error: Argument of type 'string' is not assignable to 'number'.
const x: string = add(2, 3);  // TS error: Type 'number' is not assignable to 'string'.
```

And because an async export is typed as returning a `Promise`, forgetting to
`await` it is visible to the type system:

```ts
import { hashChain } from "./native/src/lib.rs";

const digest: string = hashChain(1000);        // TS error: 'Promise<string>' is not 'string'.
const digest2: string = await hashChain(1000); // ok
```

This is the payoff of mirroring napi's real output instead of falling back to a
`declare module "*.rs"` wildcard (which would type every export as `any`).

## Fallback: `declare module "*.rs"`

If `allowArbitraryExtensions` conflicts with something in your setup, you can add
a wildcard declaration:

```ts
// rust-modules.d.ts
declare module "*.rs" {
  const exports: Record<string, unknown>;
  export = exports;
}
```

Avoid this unless forced ā€” every export becomes untyped, and you lose exactly the
argument-checking shown above.
