# Next.js example — the same Rust crate, without the plugin

> **vite-plugin-native-rust requires Vite; Next.js doesn't use Vite.** Next.js
> builds with Turbopack (default) or webpack (`next build --webpack`) — there
> is no supported path that runs Vite plugins, so `rustPlugin()` cannot be
> used in a Next.js app. This example shows the same napi-rs crate + pattern
> **without the plugin**: compile the crate with `@napi-rs/cli` yourself and
> load the `.node` addon directly from Next.js server code.

The crate in [`native/`](native/) is the standard `create-native-rust`
scaffold (same as the React Router example): a sync `add(a, b)` and an async
`sumTo(n)` that runs on napi-rs's thread pool, off the Node event loop.

- `/` — server component that renders `add(2, 3)` → **5** and
  `await sumTo(1000)` → **500500**
- `/api/rust` — route handler returning the same values as JSON, for curl

Both are `force-dynamic` so the deployed function provably executes Rust
per-request instead of baking the numbers in at build time.

## What the plugin would have automated

On Vite, `import { add, sumTo } from "./native/src/lib.rs"` is the whole
integration. Here, every piece of that had to be done by hand:

| vite-plugin-native-rust (Vite) | This example (Next.js) |
| --- | --- |
| Compile-on-demand when a `.rs` import is loaded | `predev`/`prebuild` scripts run `napi build --release --platform --cwd native` |
| Content-hash cache, rebuild only on crate changes | Cargo's incremental cache only; napi always relinks |
| Generated named-export shim that survives the bundler | Hand-written [`lib/native.server.ts`](lib/native.server.ts) with a bundler-invisible `require` |
| `.d.rs.ts` types mirrored next to the import | Types via the napi-generated `native/index.d.ts` (type-only import) |
| Addon emitted into the build + traced by @vercel/nft automatically | Manual `outputFileTracingIncludes` in [`next.config.ts`](next.config.ts) |
| Client-graph import → readable build error | Nothing stops a client import until it fails at runtime |

## The two bundler landmines (verified, not theoretical)

1. **`serverExternalPackages` does not work for local `file:` packages.**
   `native/` is wired as `"nextdemo": "file:./native"`, which npm installs as
   a symlink whose real path is outside `node_modules`. Next only externalizes
   packages that resolve inside `node_modules`, so Turbopack bundled the napi
   loader anyway and its relative `require("./nextdemo.<platform>.node")`
   broke inside the compiled chunk ("Cannot find native binding"). For a
   *published* napi package this config would have been enough.

2. **Turbopack shims `createRequire`.** The classic escape hatch —
   `createRequire(import.meta.url)` plus a dynamic path — fails at runtime
   with "Cannot find module as expression is too dynamic": Turbopack
   statically recognizes `createRequire` from `node:module` and replaces the
   returned function. The working bypass is to fetch the builtin in a way the
   bundler can't track:

   ```ts
   const { createRequire } = process.getBuiltinModule("node:module");
   const nativeRequire = createRequire(process.cwd() + path.sep);
   const native = nativeRequire(resolveNativeLoader()); // absolute path
   ```

   (`process.getBuiltinModule` needs Node ≥ 20.16 / 22.3 — Vercel runs 24.x.)

Because that require is invisible to the bundler, it is also invisible to
`@vercel/nft` tracing — `outputFileTracingIncludes` is what actually carries
`native/index.js` + `native/*.node` into the serverless function:

```ts
outputFileTracingIncludes: {
  "/*": ["./native/index.js", "./native/*.node", "./native/package.json"],
},
```

You can verify the tracing locally after a build:

```bash
node -e "console.log(require('./.next/server/app/api/rust/route.js.nft.json').files.filter(f => f.includes('native')))"
```

## Run locally

Requires a Rust toolchain (https://rustup.rs). From the **monorepo root**:

```bash
npm install
npm run dev -w example-nextjs        # predev compiles the crate first
curl http://localhost:3000/api/rust  # {"add":5,"sumTo":500500,...}
```

Production:

```bash
npm run build -w example-nextjs
npm run start -w example-nextjs
```

`native/index.js` / `native/index.d.ts` are generated by `napi build` and
committed so typechecking works without a Rust toolchain; the `.node` binary
is platform-specific and gitignored, so the build scripts always recompile it.

## Vercel deploy

Project settings (one-time, via dashboard or API): Root Directory =
`examples/nextjs`, "Include source files outside of the Root Directory"
enabled. [`vercel.json`](vercel.json) then overrides install/build:

- [`scripts/vercel-install.sh`](scripts/vercel-install.sh) — `npm install` at
  the repo root, then installs a minimal stable Rust toolchain into
  `node_modules/.cache` (the build image ships rustup with no default
  toolchain; the cache dir rides Vercel's build cache so warm builds skip it).
- [`scripts/vercel-build.sh`](scripts/vercel-build.sh) — re-exports the cargo
  PATH (installCommand env doesn't persist) and runs
  `npm run build -w example-nextjs`, which compiles the crate for
  `linux-x64-gnu` on-target and then runs `next build`.

Deploy from the monorepo root:

```bash
vercel deploy --prod --yes
curl https://vpnr-example-nextjs.vercel.app/api/rust
# {"add":5,"sumTo":500500,"node":"v24.x.x","platform":"linux-x64"}
```

At runtime the loader is resolved from `process.cwd()` (the function's cwd is
the Root Directory), with a repo-root fallback — see
[`lib/native.server.ts`](lib/native.server.ts).

## Verdict

Rust-via-napi works fine in Next.js server code — the crate is untouched and
the deployed function returns the exact expected values. What you lose without
Vite is the plugin's ergonomics: compile-on-demand with content-hash caching,
bundler-safe codegen, mirrored TypeScript types, automatic asset placement,
and the server-only import guard. All of it is reproducible by hand (this
example is the proof), but it's ~100 lines of wiring and two bundler landmines
instead of one import line.
