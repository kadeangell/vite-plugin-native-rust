# Spike findings — napi-rs runtime story (PLAN step 0)

Goal of the spike: prove the entire runtime story by hand, with **no plugin**,
so the plugin author builds against verified reality. Every checkpoint below
passed. Read the "Gotchas for the plugin author" section last — it is the
payload.

All work done on: macOS arm64 (Darwin 25.5.0), Node v25.8.1, npm 11.11.0,
cargo/rustc 1.96.0, `@napi-rs/cli` 3.7.2, Vite 6.4.3, React Router 7.18.1.

---

## The crate (kept — this is a real deliverable)

Location: `native/` at repo root (deliberately not under `app/`, so cargo's
`target/` churn stays out of the RR file watcher).

Files:
- `native/Cargo.toml` — `crate-type = ["cdylib"]`, deps `napi` v3 (features
  `napi4`, `async`), `napi-derive` v3, `sha2`, `hex`; build-dep `napi-build` v2;
  a `[profile.release]` with `lto = true` + `strip = "symbols"`.
- `native/build.rs` — `napi_build::setup();`
- `native/src/lib.rs` — the two exports (below).
- `native/package.json` — **required by the napi CLI** (see gotcha #1). Minimal:
  `{ "name": "native", "napi": { "binaryName": "native" } }`.
- `native/Cargo.lock` — generated by the first build (keep; it feeds the plugin's
  cache-key hash per PLAN D2).

Rust exports (`native/src/lib.rs`):
```rust
#[napi]
pub async fn hash_chain(iterations: u32) -> String { /* SHA-256 chain */ }

#[napi]
pub fn add(a: i32, b: i32) -> i32 { a + b }
```
`hash_chain` mirrors `app/heavy.server.ts` (same SEED `"vite-rust-import-plugin"`,
same SHA-256 re-hash loop). Verified identical output: at 700 000 iterations the
digest is `4107c82d1fa6864d261f6a5d3786d88f4814f6e1827718aee1ab096ec56b1fb3`
(stable across dev, built server, moved server, and debug vs release).

### Exact napi CLI invocation that worked

Run from **inside `native/`**:
```bash
# dev / debug (fast compile, slow runtime)
npx napi build

# production (slow compile, ~30x faster runtime, stripped)
npx napi build --release
```

Artifact locations (all relative to `native/`, the default `--output-dir` is the
crate folder):
- **`.node` binary: `native/native.node`** — filename is `<binaryName>.node`
  with **no platform triple**, because `--platform` was not passed. `file`
  reports `Mach-O 64-bit dynamically linked shared library arm64`.
  - debug: ~2.1 MB. release: ~504 KB (lto + strip).
- **`native/index.d.ts` — generated** (full content below).
- **`index.js` — NOT generated.** napi only emits its JS loader when
  `--platform` is passed. We don't need it (the plugin generates its own JS), but
  the author must know it isn't there by default.
- Cargo intermediates in `native/target/` (gitignored).

### Full generated `.d.ts` (`native/index.d.ts`)
```ts
/* auto-generated by NAPI-RS */
/* eslint-disable */
/**
 * Trivial synchronous export — exists so the plugin can exercise
 * named-export enumeration over a mix of sync/async functions.
 */
export declare function add(a: number, b: number): number

/**
 * CPU-bound SHA-256 hash chain, mirroring heavy.server.ts::heavyHashChain.
 *
 * `async` so napi runs it on the libuv thread pool: the Node event loop is
 * never blocked while this churns. Returns the final digest as a hex string.
 */
export declare function hashChain(iterations: number): Promise<string>
```
This is directly usable for PLAN step 7: copy it to `<anchor>.d.rs.ts` next to
the imported `.rs` and it types the named exports. Note `u32` → `number`,
`async fn -> String` → `Promise<string>`, and doc-comments carry through.

### Addon `Object.keys()` (ground truth for named-export enumeration)
```js
["add", "hashChain"]
```
**snake_case → camelCase**: Rust `hash_chain` becomes JS `hashChain`; `add`
stays `add`. The plugin's key-enumeration step (PLAN "Generated JS") must expect
camelCase names, not the Rust identifiers.

## Checkpoint (a) — Dev SSR shim works

Setup (temporary, since reverted): a `.server.ts` re-exporting the addon and a
`/spike` route whose loader `await`s `hashChain(700_000)` (~2.7 s in debug) and
calls `add(2, 3)`.

Result: `curl http://localhost:5173/spike` → **HTTP 200 in 2.59 s**. Rendered
body contained the digest above, `add(2, 3) = 5`, and
`addon Object.keys: ["add","hashChain"]`.

Conclusion: React Router v7 dev SSR runs a `createRequire`'d native `.node`
without complaint. The createRequire + absolute-path shim survives RR/Vite's SSR
transform intact.

### Copy-pasteable **dev** generated-JS shape (verified working)
```js
import { createRequire } from "node:module";
const require = createRequire(import.meta.url);
const addon = require("/abs/path/node_modules/.cache/vite-rust/<crate>-<hash>.node");
export const hashChain = addon.hashChain;
export const add = addon.add;
// (enumerate Object.keys(addon) in-process to emit one `export const` per key)
```

## Checkpoint (b) — Event-loop responsiveness (the whole point)

With the dev server running, fire one request, then time `/api/hello?q=x`
(trivial resource route) while the first is in flight.

| In-flight request                     | `/api/hello` latency during it |
|---------------------------------------|--------------------------------|
| **async Rust** `hashChain` (`/spike`) | **~1.1–1.3 ms** (event loop free) |
| **sync JS** `heavyHashChain` (`/slow-cpu`) | **1526 ms** (event loop blocked) |

Baseline idle `/api/hello`: ~9 ms (first hit) then ~1 ms.

Conclusion: `#[napi] async fn` runs off the Node event loop (libuv pool) — the
server stays responsive under heavy compute. The equivalent synchronous JS
blocks every other request for the full duration. This is the parallelism
contract the whole project bets on, and it holds.

## Checkpoint (c) — Build-mode emitFile + ROLLUP_FILE_URL

A temporary inline Vite plugin resolved a virtual module (imported only by the
spike `.server.ts`) and, in `load` (build + ssr only), did:
```js
const ref = this.emitFile({ type: "asset", fileName: "spike-addon.node",
                            source: readFileSync(ADDON_PATH) });
return `... require(fileURLToPath(import.meta.ROLLUP_FILE_URL_${ref})) ...`;
```

`npm run build` (`react-router build`) results:
- **`.node` is present in the server output**: `build/server/spike-addon.node`
  (2,136 kB — the debug binary we emitted).
- **What Rollup rendered `import.meta.ROLLUP_FILE_URL_<ref>` into** (from
  `build/server/index.js`):
  ```js
  const require$1 = createRequire(import.meta.url);
  const addon = require$1(fileURLToPath(new URL("spike-addon.node", import.meta.url).href));
  const hashChain = addon.hashChain;
  const add = addon.add;
  ```
  i.e. `ROLLUP_FILE_URL_<ref>` → **`new URL("spike-addon.node", import.meta.url).href`**
  — a **relative** URL resolved against the chunk's own `import.meta.url`. No
  absolute path, no knowledge of outDir. Exactly what PLAN D3 wants.
  (Aside: Rollup renamed the `createRequire` binding to `require$1` because RR's
  own entry also imports `createRequire` — harmless, just don't hard-code the
  local name.)
- Running the built server (`npm run start` → `react-router-serve
  ./build/server/index.js`, port 3000) and `curl /spike` → **HTTP 200 in 2.43 s**,
  same digest. The addon loads via the relative reference.

### Copy-pasteable **build** generated-JS shape (what to emit)
```js
import { createRequire } from "node:module";
import { fileURLToPath } from "node:url";
const require = createRequire(import.meta.url);
const addon = require(fileURLToPath(import.meta.ROLLUP_FILE_URL_<ref>));
export const hashChain = addon.hashChain;
export const add = addon.add;
```
`<ref>` is the string returned by `this.emitFile(...)`. Emit this **only** in
build mode; emit the dev shape (checkpoint a) otherwise.

## Checkpoint (d) — Moved-dir acceptance test (Sharp Edge 2)

Copied the whole `build/` to a **differently-named sibling** inside the repo
(`build-moved-spike/`) and ran the server from the copy
(`PORT=3100 npx react-router-serve build-moved-spike/server/index.js`).

Result: `curl http://localhost:3100/spike` → **HTTP 200 in 2.52 s**, same digest.
The addon loaded from `build-moved-spike/server/spike-addon.node` via the
relative reference. **No absolute path leaked** for the `.node`.

Important nuance (read before you reproduce this): keep `node_modules`
reachable. The server bundle keeps `react` / `react-router` **external** (not
bundled), so if you move `build/` completely outside the project tree, Node
fails with `Cannot find package 'react'` — that is unrelated to the `.node` and
is not what this test checks. Move to a sibling that can still resolve up to the
project's `node_modules` (or ship `node_modules` alongside, as a real deploy
would). The `.node` itself travels with the server chunk and resolves
relatively.

## Checkpoint (e) — `options.ssr` across every context

Logged `options?.ssr` in both `resolveId` and `load`. Full observed matrix:

| Context                         | `resolveId` `options.ssr` | `load` `options.ssr` |
|---------------------------------|---------------------------|----------------------|
| **dev** SSR (request to /spike) | `true`                    | `true`               |
| **build — client pass**         | `true` **and** `false` (called twice) | `false`  |
| **build — SSR pass**            | `true`                    | `true`               |

Two findings here, both load-bearing for PLAN D4:

1. **Put the D4 gate in `load`, not `resolveId`.** During the *client* build,
   `resolveId` for the same module is invoked more than once and at least one of
   those calls reports `options.ssr = true` (RR resolves the id across multiple
   environments). `load`, by contrast, reliably reports `false` in the client
   pass and `true` in the SSR pass. Gating on `resolveId`'s `options.ssr` would
   be unreliable; gating in `load` is correct. (`resolveId` may still *claim* the
   `.rs` id unconditionally — just don't make the ssr *decision* there.)

2. **`.server.ts` importers never reach the client build at all.** When the
   virtual module was imported only by a `.server.ts`, the client build never
   called `load` for it (RR strips server-only modules from the client graph).
   The `options.ssr = false` / client-pass row above was only observable via a
   deliberately *non-server* importer (a throwaway probe imported from
   `root.tsx`). This confirms the exact scenario PLAN D4 defends: the gate fires
   for a plain `.rs` import that leaks toward the client, and there it correctly
   sees `options.ssr = false` and can `this.error(...)`.

`this.meta.watchMode` cleanly distinguishes dev (`true`) from build (`false`) and
is a reliable build-vs-dev signal for choosing the dev vs build JS shape.

## Gotchas for the plugin author (the payload)

1. **napi v3 requires a `package.json` with a `napi` field in the crate dir.**
   Without it: `Internal Error: package.json not found at <crate>/package.json`.
   The crate needs `{ "napi": { "binaryName": "<name>" } }`. The plugin should
   either require the crate ship one, or generate a minimal one before invoking
   the CLI. `binaryName` determines the output filename (`<binaryName>.node`).

2. **Default output is `<binaryName>.node` at the crate root, no platform
   triple, and no `index.js`.** Only `--platform` adds the triple and emits the
   JS loader. The plugin generates its own JS, so skip `--platform` to keep the
   filename deterministic (`<binaryName>.node`) — then copy/rename it to the
   hashed cache name (PLAN D2). Use `--output-dir` to redirect output if you
   don't want it written into the crate.

3. **Gate on `options.ssr` in `load`, not `resolveId`** (checkpoint e, finding 1).

4. **Debug and release both write the same `<binaryName>.node` path** and
   overwrite each other. Dev must build debug (compile ~12 s cold here, runtime
   ~3.9 µs/iter); build must pass `--release` (compile similar, runtime ~30x
   faster: 700 k iters = **92 ms** release vs **~2730 ms** debug, and 504 KB vs
   2.1 MB). The hashed-filename scheme (D2) plus separate handling per mode keeps
   them from colliding; just don't assume one path is "the" binary.

5. **`ROLLUP_FILE_URL_<ref>` → `new URL("<fileName>", import.meta.url).href`**,
   which requires the server chunk be **ESM** (it is — RR's server build is ESM,
   so `import.meta.url` exists). `fileURLToPath(...)` around it hands
   `createRequire` a real filesystem path. This is the mechanism that makes the
   `.node` travel with the build and resolve relatively (checkpoints c, d).

6. **The moved-dir test needs `node_modules` reachable** — `react`/`react-router`
   stay external in the server bundle (checkpoint d nuance). A green moved-dir
   test only proves the `.node` path is relative; it says nothing about npm deps,
   which a real deploy ships separately.

7. **Cold cargo compile was ~12 s here** (warm dependency cache; a truly cold
   `~/.cargo` registry fetch is slower — PLAN's "~30s+" is a safe estimate). Log
   progress on first compile so dev-server startup doesn't look hung.

8. **`createRequire` local binding may be renamed by Rollup** (saw `require$1`)
   when another module in the graph also imports it. Don't hard-code or grep for
   the exact local name in tests.

9. The `require(...)` argument must stay a **static string literal / a value
   Rollup won't trace** (per PLAN). The `createRequire` + `ROLLUP_FILE_URL`
   pattern satisfies this; don't introduce a commonjs-transform plugin.
